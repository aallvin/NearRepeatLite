# NearRepeatLite

A memory-efficient implementation of the Knox test for near-repeat crime
analysis. Drop-in replacement for `NearRepeat()` from
[Steenbeck's NearRepeat package](https://github.com/wsteenbeek/NearRepeat)
for use when large datasets cause memory allocation errors.

---

## Installation

```r
# Install remotes if needed
install.packages("remotes")

# Install NearRepeatLite from GitHub
remotes::install_github("aallvin/NearRepeatLite")
```

---

## Quick start

```r
library(NearRepeatLite)

data(oslo_sim)   # synthetic example dataset (500 events, Oslo extent, 2015-2019)

set.seed(1519)
result <- NearRepeatLite(
  x    = oslo_sim$x,
  y    = oslo_sim$y,
  time = oslo_sim$date,
  sds  = c(0, 1, 101, 201, 301, 401, 501),   # spatial bands in metres
  tds  = c(0, 1, 8, 15, 22, 29),              # temporal bands in days
  nrep = 999
)

result$observed       # observed pair counts
result$knox_ratio     # Knox ratios (observed / mean simulated)
result$pvalues        # Monte Carlo p-values
```

## Background

Near-repeat victimisation describes the empirical pattern whereby crimes cluster
in space and time: locations close to an initial incident are at elevated risk
for a period following that incident. The Knox test quantifies this clustering
by comparing the observed number of crime-event pairs that are both spatially
and temporally close against a Monte Carlo null distribution generated by
randomly shuffling the time labels of events.

Steenbeck's `NearRepeat()` function is the standard R implementation of the
Knox test. `NearRepeatLite()` is not a methodological alternative — it
implements exactly the same analysis — but uses a different computational
strategy to avoid the memory constraints that crash `NearRepeat()` on large
datasets.

---

## The memory problem

`NearRepeat()` calls `stats::dist()` to compute pairwise distances between all
events. For a dataset of *n* events, this produces a condensed distance matrix
of *n(n−1)/2* values held entirely in RAM — before any binning or counting
can happen. Two such matrices are needed simultaneously (spatial and temporal),
and a fresh temporal matrix is created for every Monte Carlo permutation.

| n events | Each distance matrix | Both at peak |
|---|---|---|
| 5,000 | ~100 MB | ~200 MB |
| 10,000 | ~400 MB | ~800 MB |
| 20,000 | ~1.5 GB | ~3 GB |
| 50,000 | ~10 GB | ~20 GB |
| 79,294 (Oslo res, 5-yr) | ~25 GB | **~50 GB** |

For the Oslo residential burglary dataset pooled over five years (~79k events),
this is simply not feasible on standard hardware. R throws an allocation error
before the analysis begins.

---

## How NearRepeatLite() solves this

Instead of computing all pairwise distances at once, the function processes one
row of the distance matrix at a time. For each event *i*, distances to all
*j* > *i* are computed as a short vector, immediately binned into spatial and
temporal bands, and then discarded. Only two things are retained:

1. The **observed count** accumulated into an *ns × nt* integer matrix (tiny).
2. The **index and spatial band** of every within-range pair, stored for reuse
   across all Monte Carlo permutations.

Because most crime-event pairs in a city are separated by more than the maximum
spatial band, the within-range pair list is a small fraction of all *n(n−1)/2*
pairs. Peak RAM is therefore:

```
O(n + within_range_pairs)  instead of  O(n²)
```

For the Oslo data, this brings peak usage from ~50 GB to under 1 GB.

See [algorithm.md](algorithm.md) for a detailed walkthrough of both passes and
a proof of mathematical equivalence.

## Output

Returns an object of class `"knox"` with the same structure as `NearRepeat()`:

| Element | Description |
|---|---|
| `observed` | Contingency table of observed crime-pair counts |
| `knox_ratio` | observed / mean(simulated) for each cell |
| `knox_ratio_median` | observed / median(simulated) for each cell |
| `pvalues` | Monte Carlo p-values |
| `array_Knox` | Full 3-D simulation array (only if `saveSimulations = TRUE`) |

The output is fully compatible with `plot.knox()` from Steenbeck's package.

---

## Band definitions

Band breakpoints follow Steenbeck's convention: `right = FALSE`,
`include.lowest = FALSE`, giving left-closed, right-open intervals — e.g.
`[0, 1)`, `[1, 101)`. In the Oslo analysis, spatial breaks `c(0, 1, 101, 201, ...)`
are chosen deliberately: with Manhattan distance producing integer values, the
break at 101 ensures that distance = 100 m falls inside the "1–100 m" band
rather than the next one.

---

## Verification

Observed counts are fully deterministic and must be identical between
`NearRepeat()` and `NearRepeatLite()` for any given dataset and band
specification. An optional verification script is provided at
`inst/scripts/verify_equivalence.R`. It requires Steenbeck's NearRepeat package
and runs both functions on `oslo_sim`, confirming that all differences in
observed counts are zero.

Knox ratios and p-values will differ between separate runs due to Monte Carlo
noise. This is expected behaviour and not a sign of error.

## When to use which function

| Situation | Recommendation |
|---|---|
| n < ~15,000 | Either; `NearRepeat()` is faster for small datasets |
| n ≥ ~15,000, or R reports allocation errors | `NearRepeatLite()` |
| Replicating Steenbeck's published examples | `NearRepeat()` |
| Pooled multi-year analysis of large crime types | `NearRepeatLite()` |
